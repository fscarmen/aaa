# Product Spec

## 项目名称

qrencode-go C 版本迁移

## 项目目标

在不改变现有功能、命令行行为和输出兼容性的前提下，将当前 Go 版本二维码生成工具迁移为 C 版本实现，并补充 GitHub Actions 交叉编译工作流，支持 Linux 多架构构建与发布。

## 核心功能列表

### 1. 二维码生成

- 优先级：高
- 描述：接收命令行输入内容，生成二维码矩阵。
- 输入：命令行参数、标准输入或当前程序已支持的文本输入方式。
- 输出：与现有 Go 版本兼容的二维码结果。
- 业务规则：
  - 二维码编码结果应与现有程序行为保持一致或等价。
  - 不改变现有参数语义。
  - 不降低现有输入兼容性。

### 2. 现有渲染输出保持兼容

- 优先级：高
- 描述：保留当前终端渲染/文本渲染行为。
- 输入：二维码矩阵数据、现有渲染参数。
- 输出：与 Go 版本一致或等价的终端显示结果。
- 业务规则：
  - 保持现有输出格式兼容。
  - 保持默认渲染样式兼容。
  - 不引入破坏性格式变更。

### 3. 错误处理与帮助信息

- 优先级：中
- 描述：保留并完善命令行错误提示、参数校验与帮助输出。
- 输入：非法参数、空输入、不可处理的输入内容。
- 输出：清晰的错误信息与帮助说明。
- 业务规则：
  - 返回码应符合命令行工具习惯。
  - 错误信息应明确指出问题原因。
  - 帮助信息需覆盖现有参数和用法。

### 4. 代码结构优化

- 优先级：低
- 描述：在不改变功能的前提下，重构为清晰的 C 项目结构。
- 输入：现有 Go 项目逻辑。
- 输出：模块化的 C 源码、头文件和构建配置。
- 业务规则：
  - 优先保证兼容性，再考虑结构优化。
  - 优化不得改变用户可见行为。

### 5. Linux 全架构交叉编译

- 优先级：高
- 描述：提供 GitHub Actions 工作流，对 Linux 多架构进行交叉编译。
- 输入：源码仓库、GitHub Actions 触发事件。
- 输出：多架构 Linux 可执行文件或压缩产物。
- 业务规则：
  - 至少覆盖主流 Linux 架构。
  - 工作流应能自动构建并产出 artifact。
  - 需考虑第三方库链接方式，优先静态链接，尽量避免运行时动态依赖问题。

## 功能边界

- 不新增终端外的新 UI。
- 不新增与二维码生成无关的新功能。
- 不要求实现 Windows 或 macOS 发布工作流，本阶段聚焦 Linux。

## 技术栈建议

- 语言：C17
- 二维码编码库：libqrencode
- 构建系统：Makefile
- CI/CD：GitHub Actions
- 链接策略：
  - 优先静态链接 libqrencode 及其必要依赖。
  - 若某些架构无法完全静态链接，应在工作流和文档中明确说明。
  - 优先采用 musl 目标以减少运行时依赖问题（如适用）。

## 兼容性要求

- 命令行参数行为与当前 Go 版本保持兼容。
- 默认输出内容与格式保持兼容或等价。
- 对现有 README 中描述的主要用法保持兼容。

## 非功能要求

- 代码应清晰、可维护。
- 项目结构应适合后续扩展。
- 构建流程应可重复执行。
- 对外发布产物应尽量降低系统依赖。

## 交付物

- C 版本源码
- 头文件与构建文件
- [`Product-Spec.md`](Product-Spec.md)
- GitHub Actions 工作流文件
- 必要的构建与使用说明文档更新

## 验收标准

- 高优先级功能全部完成。
- C 版本可成功构建。
- 生成与渲染行为与 Go 版本保持兼容或等价。
- GitHub Actions 可构建 Linux 多架构产物。
- 已评估并落实库链接策略，优先静态链接。
